# 基于 micro-ROS 与异构双通道的端侧语义视觉传感器 - 进阶路线图

> [!NOTE]
> **项目定位**：作为简历的核心亮点，展示工程级复杂度的系统设计能力。
> **核心栈**：RT-Thread, C/C++ (Embedded C++), micro-ROS, 计算机视觉, AI 边缘推理。

为了完成此项目并在这个过程中深入学习 **RT-Thread** 及 **嵌入式 C/C++ 编程规范**，我们需要在系统设计时严格落实以下五大维度的具体指标。

---

## 1. 网络中间件与异构双通道通信 🌐

目标是建立两条独立、清晰且健壮的数据通路，实现语义和流媒体的异构分离。

- [ ] **低频语义流（主通道） - micro-ROS 移植**

  - 将 micro-ROS client 交叉编译并无缝链接至 RT-Thread 现有工程结构中。
  - 重写 XRCE-DDS 的底层 Transport 接口，将其桥接到基于 LwIP 协议栈的 UDP 协议上。
  - 测试并实现在局域网内稳定、低延迟地向下游节点发布结构化数据（如检测目标的 ID、坐标向量等）。
- [ ] **高频媒体流（副通道） - TCP/HTTP 推流服务**

  - 基于已有的推流器代码逻辑，在板端利用 POSIX Socket 建立轻量级的 TCP/HTTP 视频流服务。
- [ ] **纯非阻塞 I/O (Non-blocking I/O) 改造**

  - 对所有的 socket `accept`、`send` 接口采用非阻塞模式（或配合 `select`/`poll` 轮询）。
  - **红线要求**：在局域网网络波动或上位机强制断开时，绝不允许卡死板端的 RTOS 系统主循环或拖累视觉处理线程。

---

## 2. 内存管理与系统解耦 🧠

遵循嵌入式 C++ 高阶内存管控标准，消灭长航时 OOM 风险。

- [ ] **静态内存池 (Static Memory Pool) 设计**

  - **重构准则**：在推流与图像流转代码中，彻底消除诸如 `std::vector` 等会在堆上产生碎片的局部动态内存分配动作。
  - 划定一块全局的死内存（例如 BSS 段中划定 500KB 静态数组）作为专属图像缓存池，通过游标或无锁队列进行流转管理。
- [ ] **零拷贝 (Zero-copy) 数据流转**

  - 深入底层配置 MCU 摄像头的 DMA（Direct Memory Access），将捕获的外设图像帧直通搬运至指定的静态内存池。
  - 解放 CPU 算力，杜绝 CPU 傻瓜式的数据搬运（`memcpy`）产生的性能损耗。

---

## 3. 硬件加速与 AI 计算 ⚡

榨干 MCU 原生的算力护城河，让算力全部聚集于上层特征提取。

- [ ] **硬件 JPEG 编码器接入**

  - 剥离原推流代码中对臃肿封装库（如 OpenCV `cv::imencode`）的软件图像压缩算法依赖。
  - 查阅并打通 MCU 底层的硬件 JPEG 外设接口（或高度汇编优化的底层加速库），大幅度降低高频图像流压缩的 CPU 占用率。
- [ ] **端侧轻量化模型推理部署**

  - 在板端引入量化后的低算力 ONNX 轻量级视觉模型。
  - 利用零拷贝与硬解约出的 CPU 算力进行高速目标检测、分类或特征点提取。
  - **性能指标**：将核心的单帧 AI 推理侧延迟严格控制在系统时间窗口内（例如 `< 50ms`）。

---

## 4. RTOS 多线程调度与同步 ⏱️

构建基于优先级抢占的严谨并发模型。

- [ ] **严谨的优先级调度防反转**

  - **最高优先级**：网络心跳保活线程（维持 micro-ROS 连接会话不丢弃）。
  - **中优先级**：AI 推理与数据提炼线程。
  - **低优先级**：TCP 视频推流服务器与本地 LCD 硬件刷新渲染。
- [ ] **关键数据流读写竞态保护**

  - 识别“AI 线程（写入检测结果参数）”与“micro-ROS 发布线程（读取并 UDP 封包发送结果）”之间的共享数据区。
  - 结合任务频率需求，合理部署 Mutex（互斥锁）、Semaphore（信号量）或尝试实现更高级的读写锁（Read-Write Lock），防止上位机在订阅时读取到断层或被覆写的脏数据。

---

## 5. 上位机协同与验证闭环 💻

完成“端 -> 云 / 主机” 的工程级收口，展示实际集成效果。

- [ ] **ROS2 生态完美融合验证**

  - 上位机（如树莓派、Nvidia Jetson 或 Ubuntu 主机）无需额外部署转换脚本，在其原生 ROS2 环境下就能直接 `ros2 topic echo` 订阅开发板的数据流。
  - 提供样例验证代码，演示如何将开发板送上来的检测坐标汇聚进控制状态机。
- [ ] **解耦的全景可视化监控**

  - 上位机同步运行一套跨平台监控前端（可以是纯 HTML Web 或轻量 Python 脚本）。
  - 订阅 HTTP 并接收 MJPEG 高频视频流，同时解包低频 micro-ROS 数据，实现在原生视频流上“动态渲染叠加” AI 检测边界框（Bounding Box）。
  - **隔离验证**：即使强行关闭了 Web 视频推流端，底层 ROS2 节点话题吞吐的实时性与带宽必须保持绝对不受干扰。

---

> [!TIP]
> **学习建议**
> 为了实现这些目标，建议先完成 C/Cpp 面向对象线程的管理封装（如 `AppLED` 这种基建），接着逐步打通 LwIP/Socket（跑通推流），再接着适配 micro-ROS，最后将各类功能全部集成到内存池模型之上。稳扎稳打，每一次重构都要保证系统达到“微内核”级的零宏观报错状态。
